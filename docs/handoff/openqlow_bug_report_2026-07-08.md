## 🔴 重大1: レディースが「土曜14:00」と回答（正: 14:30）

**再現**（IMG_3850 / 3851）:
- ユーザー「レディース」→ AIKA「土曜**14:00**のクラスです」
- ユーザー「14:30では？」→ AIKA「申し訳ございません。レディースクラスは土曜**14:00**となっております」と**誤情報を断言し、訂正を拒否**。
- クイックリプライのボタン文言も「🙋‍♀️ 14時 レディース」。

**期待**: 「レディースクラスは土曜 **14:30** です」。

**openqlow での探し方**:
- `14:00` / `14時` / `レディース` / `ladies` を全文検索。
- canon（`src/shared/canon.ts` / `port/aika/flatup_canon.ts`）は PR #40 で 14:30 に更新済みのはず。
  → **canon 以外に 14:00 がハードコードされている場所**が残っている疑いが濃厚。
  候補: クイックリプライ定義（ボタンの label/postback）、体験予約フローのクラス時間テーブル、
  固定文言テンプレ、LINE公式側の定型文（コードでなく管理画面側の可能性も）。
- 修正後は**再デプロイ**しないと実機に反映されない点に注意（PR #40 マージ済みでも本番が古い＝未デプロイの疑い）。

## 🔴 重大2: キッズ体験のクイックリプライがクラス時間と噛み合わない

**再現**（IMG_3840 / 3848 / 3849）:
- AIKA「火曜か木曜の18:00でしたら、キッズクラスの体験ができます。どちらがご都合つきますか？」
- なのに提示ボタンは「🕙 **10〜12時**」「🕕 18〜20時」「↩ 曜日を変える」＝**曜日を聞いているのに時間帯ボタン**。
- さらにキッズは 10〜12時のクラスが無いのに「10〜12時」を選ばせ、選ぶと
  「申し訳ございません。キッズクラスは火木18:00・土13:00のみです」と**拒否**（IMG_3849）。

**期待**:
- キッズ体験の時間選択肢は、実クラス（火木 **18:00**、土 **13:00**）に限定する。
- 「曜日を聞く場面」では曜日ボタン（火曜 / 木曜 / 土曜）を出し、時間帯ボタンを出さない。
- そもそもキッズに「10〜12時」を候補として出さない（10〜12はスタッフ在館時間であってキッズクラスではない）。

**openqlow での探し方**:
- クイックリプライの定義箇所（`items` / `quickReply` / `postback` / `label`）。
- 「10〜12時」「18〜20時」を出す分岐が、**種目（キッズ/レディース/大人）に関係なく共通の在館時間**を出していないか。
  → 種目ごとに候補時間を分けるロジックが必要。
- 曜日確認ステップと時間確認ステップの**状態管理（どちらを聞いているか）**を確認。

## 🔴 重大3: 会話がループし曜日を保持できない

**再現**（IMG_3841 / 3849）:
- 「火曜か木曜どちら？」→ ユーザー「平日18〜20時で体験したい」→ また時間確認へ戻る、を繰り返す。
- ユーザーが「火曜日火曜日」「火曜日」と連打しても進行しない（曜日を認識・保存できていない）。
- 「曜日を選び直したい」→「火曜か木曜どちら？」→「平日18〜20時」→ 時間確認…と堂々巡り。

**期待**: 一度確定した「時間帯」「曜日」を状態として保持し、両方揃ったら次（名前確認）へ進む。
自由入力の曜日（"火曜日" 等）も postback と同じ扱いで受け取る。

**openqlow での探し方**:
- 予約フローのステートマシン（現在ステップの持ち方、slot filling）。
- ユーザーのテキスト（"火曜日"）を曜日スロットに束縛するパーサの有無。
- 時間スロットと曜日スロットが互いを上書きして無限ループしていないか。
